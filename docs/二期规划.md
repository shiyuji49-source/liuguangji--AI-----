# 鎏光机二期规划（明天可拍板）

> 依据：已勘验真实代码。护城河命题成立 — `src/app/api/prompt-studio/shotlist/route.ts` 直接读 `scriptEpisodes.content`，`buildShotlist`（`src/lib/prompt-studio/run.ts:274`）产出带 `assetRefs` 的 `shots` 行，"写完即下沉分镜/@资产"的数据管线大半已通。成本现状已核：`.env` 里 `LLM_MODEL_HEAVY` 与 `LLM_MODEL_MAIN` **双档都跑 kimi-k2.6**（opus/sonnet 已注释），所有"剧本类调用成本低"的估算必须按 kimi 实价（输入 650/M、输出 2700/M）重算。

---

## 1. 二期整体愿景（一句话）

**把鎏光机做成"剧本进、成片出"的短剧制作总台：剧本工作台（源头/stage0）写完即一键下沉分镜 → 提示词生成器产出结构化提示词 → 图像生成器把提示词烧成入墙的 @资产母版图 → 视频生成器把 @资产 + 静帧锚喂给 Seedance 出片 —— 四块共享同一项目数据流和资产体系，每一环的产物都是下一环的强制上下文，别人写完只能导 PDF，我们写完直接拍。**

四块的拼接关系（数据流，非并列功能）：

```
剧本工作台(stage0)  →[scriptEpisodes/故事圣经]→  提示词生成器(stage1-4)
                                                      ↓ prompt_items(资产/静帧/视频)
图像生成器  ←[读 prompt_items.promptText]←──────────┘
   ↓ 出图入 assets 表(@名/母版图/filePath)
视频生成器  ←[@资产母版图 + shots.stillPrompt 静帧锚]
   ↓ gen_tasks(video) → 成片
```

唯一的跨应用耦合点是 **artifacts / prompt_items / assets 三张共享表**（产品铁律：禁止应用间直接 import 业务码，只经数据表交接）。这是设计约束，不是妥协。

## 2. 四块详规

### 块 A — 剧本工作台（剧本医生完整版，stage0 源头）

| 维度 | 内容 |
|---|---|
| **定位** | 从"孤立诊断工具"重定位为"流水线源头工作台"。AI 是**旁路 diff 建议器**，不是对话框。产物必须落成结构化对象。 |
| **MVP 范围**（砍到三件） | **A1 故事圣经字段化 + 导入反向抽取**：新增 `story_bible` 表（题材/基调/世界观/核心人设关系/付费点策略/节奏模板），复用现有"上传即自动分集"链路反向抽取，**抽取产物落 artifacts 表、结果可人工编辑校对**（抽取准确率是体验命门）。强制注入 `buildRuntimeNote`（`src/lib/ai/skills.ts`），让现有诊断/改写/分镜 stage 立刻更准。**A2 诊断结构化**：把 `src/lib/script-doctor/run.ts` 现在的只读 Markdown 升级为 JSON（维度评分 + 场号定位 + 每条带"一键应用"建议）。**A3 改写改 diff**：新增 `script_episode_snapshots` 表，改写前存命名快照，结果**段/句级** diff、逐条接受/回滚（先不做逐词级）。 |
| **明确不进 MVP（下放 P1）** | 无限画布 Beat Board、人物关系图谱、时间线/戏量视图、集级任意两版对比、逐词级 diff。 |
| **标杆借鉴** | Sudowrite Story Bible（结构化字段强制注入上下文 + Import Novel 反向抽取）；ProseEngine（命名快照 + 逐条接受/回滚 = diff 标准形态）；Novelcrafter Extract from Chat（对话只是输入口，产物必须 Extract 回结构化）；海马轻帆（诊断对标，但国内止步导 PDF，我们诊断完直接进分镜）。 |
| **融合方式** | `story_bible` = 项目级配置自然延伸，复用 `projects` 表 + 现有规格下沉机制；分集结构挂在已有 `scriptEpisodes` 上；诊断/改写/抽取都是 LLM 调用，复用 `chargeLlm`（`reason='llm'`）。**run.ts 现已是单 system 块，新增功能必须延续单块合并**（MEMORY: Kimi 多 system 块丢 skill）。 |
| **交互要点** | 选中集/场 → 结构化诊断卡（定位到行 + 一键应用）；改写前自动快照 → 段级 diff → 逐条接受/回滚；refine 对话仅作输入口。**铁律：所有 AI 产物可接受/可回滚，否则退化成聊天框。** |
| **计费** | 走现有 `llm` 档。**按 kimi 实价重算**：诊断/改写都是长上下文重调用（`MAX_OUT=16000`，喂 100k 字原文），单次输出可达万 token，按 2700/M 输出 → 单次改写可能数十积分，不是"成本低"。`cached_in_ratio=0.1` 的降本假设不可靠（缓存命中不稳）。 |
| **技术选型** | diff 用 `jsdiff`/`diff-match-patch`（无新 UI 库，符合铁律）；JSON 诊断沿用现有 `parseJsonArrayLoose` 宽容解析 + 截断恢复（Kimi 长输出会截断已证实）。短剧方法论**不重写 skill** — 已在 `script-revision/references/micro-drama-craft.md`（钩子/反转/爽点/付费点全覆盖）和 `ai-adaptation.md`/`filmability.md`（合规），至多补一份诊断 output schema 说明。 |
| **改造量** | **A1 = M**（新表 + 抽取链路复用 + 可编辑 UI）；**A2 = M**（JSON 化 + 定位 + 应用按钮）；**A3 = M**（快照表 + diff 组件手搓）。整块 **M**（不含下放 P1 的画布/图谱，那些是 L）。 |

## 3. 开发顺序与里程碑

**核心判断：先打通"写完即下沉 + 出图入墙"的纵向闭环价值，再扩可视化。块 B 是依赖枢纽（C 强依赖 B 的母版图）。**

| 里程碑 | 内容 | 依赖 | 价值 |
|---|---|---|---|
| **M1（先做，2-3 周）** | 块 A 的 A1+A2+A3（故事圣经字段化 + 诊断结构化 + 改写 diff） | 无（纯改 LLM 旁路 + 新表） | 投入小、感知强、立刻让现有诊断/改写更准；验证"AI 旁路 diff"反对话框范式是否成立 |
| **M2（核心，3-4 周）** | 块 B 图像生成器全量 MVP（含融图，不含 mask） | 无强依赖（可与 M1 并行，但建议串行先验证计费范式） | **打通"提示词→出图→入墙→@资产"闭环，这是平台从"产文本"到"产资产"的质变**，最能消解工具碎片化痛点 |
| **M3（4-5 周）** | 块 C 视频生成器 MVP + 块 B mask 局部重绘 | **强依赖 M2**（必须有母版图作锚） | 末端交付，闭合"剧本→成片" |
| **M4（P1 后段，按需）** | 块 D：Beat Board + 关系图谱 + 时间线；块 A 的逐词 diff/任意两版对比 | 依赖 M1 的故事圣经数据 | 可视化辅助，锦上添花，工程量 L，不阻塞闭环价值 |

**关键依赖链：B(母版图) → C(视频锚)。A 与 B 可解耦并行，但建议串行：先用 A 验证旁路 diff + Kimi 长上下文计费范式，再上 B 的 image provider 计费范式。**

## 4. 需老板明天拍板的决策点

**决策 1 — 上线时主模型用谁？（成本/质量地基，影响所有 LLM 计费估算）**
- 现状：`.env` 双档全跑 kimi-k2.6，opus/sonnet 已注释，但 `defaults.ts` 仍留 sonnet/opus 价档。
- A) 全量 kimi-k2.6（输入 650/输出 2700/M），最省；B) 剧本医生/诊断走 opus（质量优先，贵 5-7 倍），其余 kimi；C) 等图像/视频上线后按实测留存数据再定。
- **推荐 A**：二期 LLM 调用量随诊断/改写/diff 暴增，先吃 kimi 成本红利；预留 `LLM_MODEL_HEAVY` 可随时切 opus 做高价值集。**但务必按 kimi 实价重做一遍二期积分定价表**，别拿"纯文本成本低"拍脑袋。

**决策 2 — 二期范围是"纵向闭环"还是"横向铺四块"？**
- A) 纵向：A→B→C 闭环先行，无限画布(D)下放 P1；B) 横向：四块各出半成品同时推。
- **推荐 A**：横向铺四块会一次性引入 diff/画布/图谱/时间线四套手搓 UI（全是 L），且违反"先验证闭环价值"原则。D 明确下放。

**决策 3 — "一键下沉分镜"的实现边界（防与 prompt-studio 重叠 + 防违反铁律）**
- A) 在剧本工作台里复用调现有 `buildShotlist`（经接口/artifacts 交接，不重写）；B) 在剧本工作台里重写一份分镜生成。
- **推荐 A**：L3"一键下沉"≠新功能，它就是 prompt-studio 的 stage2。重写会一份剧本两处抽资产、@名清单不一致，且违反铁律 2（应用间只经 artifacts 交接）。**故事圣经抽取只产上下文对象，资产提取仍归 prompt-studio。**

**决策 4 — 出图/视频结果的对象存储方案？（B/C 都卡这）**
- A) 服务器本地磁盘（最快上线，单机风险）；B) 对象存储（OSS/S3，需选型配额）；C) 直接存 provider 返回的临时 URL（会过期，不可靠）。
- **推荐 B**：资产母版图是核心资产要长期可复用，临时 URL（C）会过期导致 @资产失效；本地磁盘（A）可作内测过渡，但上线前必须迁 OSS。`assets.filePath/thumbPath` 已预留。

**决策 5 — 合规审核做成"诊断必跑维度"还是"可选项"？**
- A) 必跑维度（每次诊断强制输出合规评分，出海前尤其）；B) 可选开关。
- **推荐 A**：短剧政治敏感/成人内容是上线红线，方法论已在 `ai-adaptation.md`/`filmability.md`，成本是把它写进诊断 output schema，几乎零增量。

## 5. 风险与未知数

1. **做成对话框（首要风险）**：当前 `script-doctor` 已有 refine 输入框 + 只读 Textarea。若 A2/A3 不强制把 AI 产物落成可接受/回滚的结构化对象，就退化成聊天框、丧失下沉价值。**这是块 A 的成败线。**
2. **成本被误判**：双档都是 kimi（非 opus/sonnet），诊断/改写/diff 都是长上下文重调用（`MAX_OUT=16000`，喂 100k 原文）。叠加 Kimi 三坑（多 system 块丢 skill→必须单块合并、缓存命中不稳→`cached_in_ratio=0.1` 降本假设不可靠、token 计数不一致→charge.ts 按实测 usage 盯死）。"纯文本成本低"过于乐观，**二期定价必须按 kimi 实价重算并实测验证**。
3. **抽取准确率决定体验**：导入剧本反向抽故事圣经若不准且不可编辑，会污染所有下游强制上下文（诊断/改写/分镜全错）。**必须人工校对兜底 + 抽取结果可编辑**，否则一错错到底。
4. **@资产母版图 inline 预览被高估**：现状 @ 机制是纯字符串匹配（`assetRefs` 是文本数组，`knownAssets` 仅作 LLM 提示），`assets` 表（带 filePath 母版图）目前为空、是 P1 净新增。所以"母版图 inline 预览"**依赖块 B 先把资产母版图落库**，不是"复用现有机制、改造量中"。块 A 的 @ inline 预览要么等 B，要么先做纯文本 @ 联动。
5. **工程量低估**：diff 渲染、关系图谱、时间线、Beat Board、视频时间线 —— `src/` 零相关代码，CLAUDE.md 禁引新 UI 库（全在 shadcn 上手搓）。研究里标"改造量中"的可视化项明显偏轻，**整体下放 P1 并接受更高估时**。
6. **跨集一致性硬问题**：长剧多集 @资产一致性是 Sudowrite/Novelcrafter 都在解的难题。必须强制注入故事圣经 + 已登场资产，避免人设崩塌（短剧刚需）。这是块 C 视频出片质量的天花板，且没有银弹。
7. **视频是分钟级长任务**：块 C 不能用块 B 的短轮询，需 webhook 或长轮询 + 失败重试 + 退费（`reason='refund'` 已预留），否则用户体验和计费都会出问题。
