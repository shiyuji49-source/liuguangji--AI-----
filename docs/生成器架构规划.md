## 1) 一句话定位

**生成器是"把提示词生成器吐出的文本，变成真图/真视频，并回灌到项目资产墙"的执行末端**——提示词生成器(P0)产出资产/静帧/视频三类提示词文本 → 图像生成器(P1)/视频生成器(P2)接收这些文本+@素材参考图，调外部引擎出图/出片 → 成品落 `assets` 表(带 `kind/atName/directorApproved`)进资产墙，从此可被项目里其它环节 @ 调用。即:**提示词生成器=写剧本，生成器=拍摄，资产墙=片库**,三者已由现成的 `artifacts`/`assets` 表串联,生成器不需要新拉关系。

### 诚实结论:"整站换壳" vs "借 UI + 自接 API"

**对我们,"借 UI + 自接 API" 是唯一现实路径,不要整站换壳。** 理由:
- 我们的后端已经长好了——`charge.ts`(铁律计费内核,零改)、`genTasks`/`assets`(已建表已迁移,零改)、`registry`(P1/P2 槽位+权限+isAppLive 闸门,微改)、`llm.ts` 的"裸 fetch+Bearer+env 路由"provider 范式(可仿)。换壳仓库的后端(Google SDK / muapi 网关 / Replicate / Stripe)**全部要删**,它们没有我们的积分铁律、没有项目/角色体系、没有资产墙概念。
- 我们要的只是**前端工作台的布局与交互状态机**(参考槽、档位选择器、提交→轮询、结果网格、骨架屏、lightbox)。openjourney 因栈同构(Next15+TS+shadcn)能成块手抄进我们的手搓 shadcn 体系。
- 一句话:**抄 openjourney 的 JSX/状态机做前端骨架,后端全用我们自己已预留的钢筋。**

## 3) 档位 → 参数 → 计费 映射表

> 关键纠偏已并入(见脚注)。所有外部调用**必经 charge.ts**;`providerCostEst` 按真实成本算,售价档位是我们自家加价梯度。

### 图片(两引擎共用 UI 三档:低/中/高)

| UI 档 | image2 `size` | nano banana(Gemini)尺寸 | 成本口径(真实) | 现行售价(defaults.ts) |
|---|---|---|---|---|
| 低 | `"1K"` | 1K | **按张恒定**(与分辨率无关,失败不收费) | `image.per_1k`=24 积分/张 |
| 中 | `"2K"` | 2K | 同上 | `image.per_2k`=36 积分/张 |
| 高 | `"4K"`(seedream4.0/4.5);nano 4K | 4K | 同上 | `image.per_4k`=48 积分/张 |

- **image2 调用要点**:`POST .../api/v3/images/generations`,`model=doubao-seedream-4-0-250828`(做成可配 env,别写死),`size` 用大写 `"1K"/"2K"/"4K"`,参考图传 `image:[url|dataURL]`(≤10 张),`watermark=false`(商用),`response_format="url"`,**同步返回**(client 超时 ≥60s)。
- **nano banana(Gemini 3 Pro Image)**:`POST generativelanguage.googleapis.com/v1/models/gemini-3-pro-image:generateContent`,同步。**国内接入是明天的拍板点**(见第 7 节)。
- **纠偏(成本端)**:真实火山是**每张一口价**,不按分辨率乘。`per_1k/2k/4k=24/36/48` 保留为**我们的售价梯度**(给用户的清晰度溢价),但 `providerCostEst = 出图张数 × 单张恒定成本`,**严禁按分辨率乘**。`image.input_per_1m=5300` 这项(按输入 token 计成本)**上线前到方舟控制台核**——若图像确为纯每张价,此项可能不成立,届时在 admin 定价表归零即可(无需改码)。

### 视频(Seedance 2.0,单引擎)

| UI 档 | Seedance `resolution` | 是否支持 | 成本口径 | 售价键(defaults.ts) |
|---|---|---|---|---|
| 标清 | `"480p"` | 标准/fast 均支持 | token 计费 | **需补** `video.480p.per_1k_tokens` |
| 高清 | `"720p"` | 标准/fast 均支持 | token 计费 | `video.720p.per_1k_tokens`=6.9 |
| ~~超清~~ | ~~`"1080p"`~~ | **❌ 标准与 fast 均不支持** | — | **删 `video.1080p.per_1k_tokens`=7.65** |

- **Seedance 调用要点**:create `POST .../api/v3/contents/generations/tasks` → 返回 `{id: cgt-xxx}`;轮询 `GET .../tasks/{id}` 看 `status: queued→running→succeeded/failed`;成功后从 `content.video_url`(签名 URL,~24h 过期,**必须立刻转存我方存储**)取片。字段:`resolution`/`ratio`(七档含 adaptive)/`duration`(**下限 4 秒,区间[4,15],可能离散 4/5/6/8/10/12/15——核控制台**)/`generate_audio`/`return_last_frame`/`seed`,fps 固定 24。参考图与首尾帧走 `content[]` 数组靠 `role` 区分(`first_frame`/`last_frame`/`reference_image`)。
- **纠偏(致命)**:**Seedance 2.0 没有 1080p**(标准与 fast 都封顶 720p)。所以:① UI 视频档位**只放 480p/720p 两档**;② `defaults.ts` 的 `video.1080p.per_1k_tokens=7.65` 是给**不存在的清晰度**定价,**删除**,新增 `video.480p.per_1k_tokens`;③ 原 1080p 计费公式 `tokens=宽×高×帧率×秒÷1024` 与官方公布的 patch token 量对不上(官方按 patch/压缩计数,不是裸像素),**结算一律以方舟返回的实际 `usage` 为准**,公式只用于 precheck 上限估算。
- 成本基准:不含视频输入 ¥46/百万 token、含视频输入(V2V) ¥28/百万 token,约 ¥1/秒(720p)。**上线前登录方舟控制台核当日官方价。**

## 4) 数据 / 任务架构

### 流转(图片=同步收口、视频=异步轮询,两条但共表)

```
[提交]  用户在工作台选档位+提示词+参考图
   │  requireProjectMember 校验(auth-helpers.ts:44)
   ▼
[precheck] estimateImage/VideoMaxCredits 按档位估上限 → charge.precheck(余额不足→402)   ← 异步任务必须"提交前"扣检
   ▼
[落库] INSERT genTasks(appKey, status=queued, input jsonb, createdBy, projectId)
   ▼
┌─ 图片(同步) ──────────────┐   ┌─ 视频(异步) ─────────────────────┐
│ claim queued→running(原子 UPDATE)│   │ claim → running,POST create → 存 providerTaskId(cgt-xxx)│
│ POST images/generations(等结果)  │   │ 后台轮询 GET tasks/{id} 直到 succeeded/failed         │
│ 拿 data[].url/b64               │   │ succeeded → 立刻把签名 video_url 转存我方存储          │
└──────────┬──────────────┘   └──────────────┬────────────────┘
           ▼                                    ▼
[转存] storage.put(buffer) → filePath(+thumbPath);二进制由 GET /api/assets/[...path] 鉴权吐字节
           ▼
[结算] chargeImage/chargeVideo 按真实用量 applyCredits(reason image|video, ref:{taskId})  ← 失败不收费/部分退款
           ▼
[入墙] INSERT assets(kind,atName,filePath,thumbPath,meta:{prompt,params,sourceTaskId},directorApproved=false)
       UPDATE genTasks(status=succeeded, resultAssetIds[], costCredits, providerCostEst)
```

- **轮询驱动(明天拍板点之一)**:Seedance 是异步,选**前端轮询 `GET /api/.../tasks/[id]`** 还是**后台 worker**。MVP 推荐**前端轮询 + 服务端"懒推进"**(每次前端 GET 时服务端顺手 poll 一次方舟并回写状态机),不引队列、不引 cron,最短路径。webhook 火山支持但增加公网回调复杂度,MVP 不做。
- **并发 claim**:复用现成抢占式范式 `UPDATE genTasks SET status='running' WHERE id=? AND status='queued' RETURNING`(仿 `items/[id]/generate/route.ts:41-48`),防双击重复提交。

### 存储方案(最大缺口,从零)——新建 `src/lib/storage/`

现状:`STORAGE_DRIVER=local`/`STORAGE_LOCAL_DIR=/data/assets` 纯占位,src 零引用,无 next.config rewrites/headers,无图像库(无 sharp)。需建:
- `src/lib/storage/index.ts`:`put(buffer,{key,contentType})→{filePath,publicUrl}`、`getStream(key)`,driver 抽象(local 先行,预留 OSS/S3)。
- **缩略图**:依赖无 sharp。MVP 二选一(拍板点)——(a) 加 sharp 生成 `thumbPath`;(b) `thumbPath` 暂存原图 URL/同图,前端 CSS 缩放。推荐 **(b) 先上线,sharp 后补**(省一个 native 依赖)。
- **静态服务**:`/data/assets` 不在 public。新建 `GET /api/assets/[...path]` 鉴权后 `new Response(stream)`(仿 `export/route.ts:200`)吐字节,资产 `filePath` 存逻辑 key。

### charge.ts 新增(仿 `chargeLlm`,M)

- `calcImageCostCredits({resolution, count})`:`p["image.per_"+res] × count`(成本端按张恒定,售价端读档位),`Math.ceil`。
- `calcVideoCostCredits({resolution, usageTokens})`:`usageTokens/1000 × p["video."+res+".per_1k_tokens"]`,**结算用方舟实际 usage**。
- `chargeImage/chargeVideo(opts)`:内部 `applyCredits({reason:"image"|"video", ref:{appKey,taskId}})`——`LedgerReason` 已含 `image|video`(schema.ts:342),**零改 schema/seed**。
- `estimateImage/VideoMaxCredits`:供 precheck 用,**异步任务提交前按档位估上限扣检**(铁律:避免轮询完才发现余额不足)。
- **失败/退款边界**:图片失败不收费(同步,直接不扣);视频 `failed/expired/cancelled` 时若已 precheck 占用则全额退(applyCredits 正 delta, reason `refund`)。

## 5) UI 形态(三栏布局,手搓 shadcn)

**图像工作台**(`src/apps/image-studio/`):
- **左栏-输入**:引擎切换 Tab(image2 / nano banana pro,**只此两个,无模型下拉**) → sticky 提示词框(可"从提示词生成器带入") → 参考图上传槽(拷 sadmann7,多图≤10,支持 @素材选择) → 档位选择器(低/中/高=1K/2K/4K) → 生成按钮(显示预估积分)。

**视频工作台**(`src/apps/video-studio/`,P2,复用 P1 骨架):
- 左栏档位换成 `resolution(480p/720p)` + `ratio` + `duration(4-15s)` + `generate_audio` + 首尾帧/参考图/参考视频槽(role 区分);

- 右栏入墙同图片(`kind="视频"`)。

接入点(微改):`registry.ts` 把 `CURRENT_PHASE: "P0"→"P1"` 即点亮图像生成器(零改 registry);`[appKey]/page.tsx` switch 新增 `image-studio`/`video-studio` 两个 case 分发组件(已有 isAppLive + 角色双校验)。

## 7) 明天拍板决策点

1. **nano banana(Gemini 3 Pro Image)国内接入方式** — 选项:(a) 走我们已有的乐奇 OpenAI 兼容层(`OPENAI_BASE_URL=https://lqapi.top/v1`,若乐奇代理了 Gemini Image,一个 key 统一)/(b) 火山方舟若上架 Gemini-via-ARK 则复用 ARK_API_KEY/(c) 直连 Google 官方端点(国内需代理,运维重)。**推荐 (a) 优先**——已有 key、已有 provider 范式、零新增公网出口;明天先确认乐奇是否代理了 gemini-3-pro-image。

2. **存储选型** — 选项:(a) local driver(`/data/assets`+鉴权路由)/(b) 直接上 OSS/S3。**推荐 (a) MVP 先 local,driver 抽象预留 OSS**——最短路径,上量再切。

3. **缩略图** — 选项:(a) 加 sharp 真生成缩图/(b) thumbPath 暂用原图 CSS 缩放。**推荐 (b)**,省 native 依赖,上线后看带宽再补 sharp。

4. **视频轮询驱动** — 选项:(a) 前端轮询+服务端懒推进/(b) 后台 worker/cron/(c) webhook。**推荐 (a)**,不引队列/cron,最短路径。

5. **是否做画布(对比滑块/瀑布流)** — 选项:(a) MVP 不做画布,只做结果网格+lightbox/(b) 上对比滑块+虚拟瀑布流。**推荐 (a) 先不做**;对比滑块零依赖可后补手搓,瀑布流量大再借 masonic 思路。**masonic 是否破例引库需你点头**——默认不破例,手搓。

6. **defaults.ts 计费校正(上线前必做,可在 admin 改不必改码)** — (a) `video.1080p.per_1k_tokens=7.65` **删除**,新增 `video.480p.per_1k_tokens`;(b) `image.input_per_1m=5300` 待方舟控制台核实图像是否纯按张计费,若是则归零;(c) 三档图片售价(24/36/48)确认保留为售价梯度,`providerCostEst` 按张恒定。**推荐:明天定方向,数值上线前登方舟控制台核当日官方价后在 admin 定价表落定。**
